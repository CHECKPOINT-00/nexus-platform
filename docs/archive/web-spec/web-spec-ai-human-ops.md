# AI, supporter, dashboard, và giao tiếp case  ## Quyết định chốt  Nexus không phải AI-only.  MVP phải là **AI-assisted + supporter-controlled**.  AI làm phần chuẩn hóa, phân tích nháp, hỏi làm rõ, tổng hợp, và phát hiện thiếu dữ liệu.  Supporter kiểm tra, chỉnh, và chịu trách nhiệm trước khi output quan trọng đi tới user.  ## AI nên làm gì  - hỏi intake có cấu trúc; - phát hiện field thiếu; - phân loại vấn đề; - tạo audit draft; - gợi ý lỗi nghiêm trọng và lỗi nên sửa; - chuẩn bị bản nháp report; - tóm tắt case cho supporter.  ## Supporter nên làm gì  - đọc case được giao; - review AI draft; - chỉnh nội dung và logic; - hỏi user khi thiếu thông tin; - quyết định report final; - theo dõi revision và re-audit.  ## Report lifecycle  Report không được là text AI gửi thẳng cho user.  Report phải đi qua lifecycle:  `AI_DRAFT` → `SUPPORTER_REVIEW` → `APPROVED` → `SENT` → `SUPERSEDED` hoặc `ARCHIVED`  ## Dashboard và workspace  ### Supporter workspace  Supporter cần một hàng đợi riêng để làm việc hằng ngày.  Màn hình tối thiểu:  - danh sách case được assign; - trạng thái; - deadline; - package; - latest version; - flags; - next action.  Nên có thêm cột/field hay filter cho:  - checkpoint; - payment status; - review round; - last activity; - urgent; - waiting user; - waiting supporter; - missing data.  Supporter cần mở case và thấy ngay timeline, tài liệu, draft, report, và message liên quan.  Case workspace nên đủ 3 vùng:  - left: timeline/lifecycle; - middle: tài liệu, report, draft; - right: message, note, next action, meeting/payment context.  ### Ánh xạ UI dashboard với HeroUI v3  Các component HeroUI v3 dùng cho dashboard và workspace:  - **Table**: dùng cho danh sách case ở cả supporter và admin — hỗ trợ structured data, sorting, selection, column resizing, infinite scrolling. - **Filter/Select**: dùng để lọc case theo trạng thái, package, supporter, deadline. - **Badge/Chip**: gán package, severity tag, trạng thái payment cho từng case. - **DatePicker/Calendar**: dùng để chọn deadline CP, chọn ngày hẹn supporter, chọn khoảng ngày cho DateRangePicker. - **Modal/Drawer**: mở chi tiết case, assign supporter, xác nhận payment. - **Tabs**: chuyển giữa các view trong workspace case (tài liệu, report, timeline, message).  HeroUI v3 xử lý UI layer. Workflow logic, permission, và data flow nằm ở backend Hono.  ### Admin console  Admin cần nhìn hệ thống ở mức vận hành:  - assign supporter; - payment; - data quality; - queue; - case lifecycle; - workflow config; - analytics cơ bản.  Admin cũng cần thấy assign queue, payment verification queue, và case health để điều phối supporter.  ## Giao tiếp trong case  MVP không nên làm realtime chat phức tạp.  Kênh chính là case-based message, comment, và event log.  ### Case messages  Message tự do chỉ nên gắn với `case_id`.  Không nên ép mỗi message phải mang nhiều metadata.  Metadata kiểu document/report/audit round nên sinh từ action có cấu trúc hoặc case event, không đoán từ chat text.  ### Case events  Các hành động có cấu trúc mới tạo event, ví dụ:  - upload file; - submit revision; - gửi report; - xác nhận payment; - tạo meeting; - approve report.  Đây là nơi hợp để lưu trace như ai upload, ai review, và object nào được tạo mới.  ### Entity comments  Nếu sau này cần comment riêng cho report hoặc document, tách thành feature riêng.  ## API và workflow core  Workflow lõi nằm trong Hono backend, không đặt trong n8n.  Vercel AI SDK dùng trong Hono để gọi model và stream output.  n8n chỉ để automation phụ như mail, notification, hoặc nhắc lịch.
